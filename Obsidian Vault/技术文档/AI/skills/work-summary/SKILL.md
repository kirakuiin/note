# 工作总结 Skill

你的任务是将用户与你交互的对话内容提炼为结构化的工作报告，帮助用户快速产出日报、周报、月报、年报，并保证追加写入后文档仍像一个完整整体。

## 前置依赖：Obsidian Markdown

**在生成任何报告之前，必须先使用 `use_skill` 加载 `obsidian-markdown` skill**，以获取 Obsidian Flavored Markdown 的完整语法规范。所有生成的报告都必须遵循该 skill 提供的语法标准，包括但不限于：

- **YAML frontmatter**（properties）：每个文件必须以 `---` 包裹的元数据开头
- **Wikilinks**：使用 `[[文件名]]` 语法在日报、周报、月报之间建立双向关联
- **Callouts**：使用 `> [!type]` 语法高亮关键信息（反思、问题、亮点等）
- **Tags**：使用 `#tag` 和 `#nested/tag` 进行分类和检索
- **高亮**：使用 `==高亮文本==` 标记重点内容

## 🔧 文件操作规范（必须遵守）

> **重要**：为避免 AI 模型误判文件状态和编码问题，本 skill 规定了标准的文件操作流程。

### 1. 文件存在性检测

**禁止使用 AI 内置的文件判断**，必须通过终端命令检测：

```bash
python -c "import os; path=r'<目标路径>'; print('EXISTS: file' if os.path.isfile(path) else ('EXISTS: dir' if os.path.isdir(path) else 'NOT_EXISTS'))"
```

**使用场景**：
- 写入日报前检测文件是否已存在（决定是覆盖还是合并）
- 检测目录是否存在
- 生成周报前检测本周各日报是否存在

### 2. UTF-8 编码写入（临时文件流程）

由于 AI 的 `edit_file` 工具在某些项目环境下默认使用 GBK 编码，且已存在的 UTF-8 文件不能用 GBK 方式处理，**必须使用临时文件流程**确保编码正确：

#### 统一流程（新建和追加都适用）

```
步骤 1：如果目标文件已存在，用 read_file 读取原内容（UTF-8）
步骤 2：AI 合并新旧内容，生成完整文档
步骤 3：用 edit_file 写入临时文件（如 _temp_daily.md，与目标文件同目录）
步骤 4：用 Python 将临时文件（GBK）转为目标文件（UTF-8）
```

#### 步骤 4 的 Python 命令

```bash
python -c "import os; src=r'<临时文件路径>'; dst=r'<目标文件路径>'; content=open(src,'r',encoding='gbk').read(); open(dst,'w',encoding='utf-8').write(content); os.remove(src)"
```

**完整示例**：

```bash
# 假设临时文件和目标文件路径如下：
python -c "import os; src=r'C:\Users\wangzhuowei\note\Obsidian Vault\netease\daily\2026\03\_temp_daily.md'; dst=r'C:\Users\wangzhuowei\note\Obsidian Vault\netease\daily\2026\03\2026-03-30_日报.md'; content=open(src,'r',encoding='gbk').read(); open(dst,'w',encoding='utf-8').write(content); os.remove(src)"
```

#### 为什么用临时文件？

| 问题                | 临时文件方案如何解决                 |
| ----------------- | -------------------------- |
| edit_file 默认 GBK  | 临时文件用 GBK 读取，转为 UTF-8 写入目标 |
| 已存在的 UTF-8 文件会被破坏 | 不直接修改目标文件，而是覆盖写入           |
| 命令行长度限制           | 内容在文件中，不受命令行长度限制           |

#### 临时文件命名规范

- 固定前缀：`_temp_`
- 放在目标文件同目录
- 示例：`_temp_daily.md`、`_temp_weekly.md`

### 3. 目录创建

如果目标目录不存在，使用：
```bash
mkdir "<目录路径>"
```
或在 PowerShell 中：
```powershell
New-Item -ItemType Directory -Path "<目录路径>" -Force
```

### ⚠️ 强制执行规则

1. **先检后写**：写入任何文件前，必须先通过终端命令检测文件是否存在
2. **已存在文件先读取**：如果目标文件已存在，用 `read_file` 读取原内容后合并
3. **临时文件流程**：
   - 第一步：用 `edit_file` 写入临时文件（`_temp_xxx.md`）
   - 第二步：用 Python 命令将临时文件（GBK）转为目标文件（UTF-8）并删除临时文件
4. **禁止直接修改已存在的 UTF-8 文件**：必须通过临时文件流程，否则会导致乱码

## 核心原则

- **精干简练**：不要堆砌细节，每条工作项用 1-2 句话说清做了什么、解决了什么问题
- **准确反映**：忠实于实际对话内容，不捏造不夸大，用户提到的问题和解决方案都要体现
- **合理估时**：根据对话复杂度和工作量给出合理的工时估算（单位：小时，精确到 0.5h）
- **文档整体性**：追加更新时必须先读取既有内容，做结构对齐、语气统一、条目合并与去重，保证新增后读起来像一篇完整文档而不是拼接日志
- **Obsidian 原生体验**：生成的文件在 Obsidian 中应充分利用双链、标签、callout 等特性，便于知识管理和回顾

## 通用实现细则（追加更新必须遵守）

1. **先读后写**：写入前必须读取目标文件完整内容。
2. **结构检测**：识别 frontmatter、主标题、工作内容区、callout 区、统计区是否存在。
3. **条目续号**：若已有 `### N. `，新增编号从最大 N+1 开始。
4. **合并与去重（强制）**：
   - 与已有条目语义相同（同一任务仅措辞不同）时，不新增重复条目，改为补充“进展/结果”。
   - 同一问题的后续处理应合并到原条目“问题与解决”中，而不是新建重复问题。
   - 同一项目标签重复出现时只保留一份。
5. **统一文风**：新增条目使用与既有内容一致的粒度、句式、标题层级。
6. **统计回写**：更新 frontmatter 的 `total_hours` 与文末“当日/当周/当月合计”。
7. **最小扰动**：不重排无关历史内容，不因“美化”大改旧文结构。

## 日报生成

### 触发条件
用户明确要求日报，或在中触发确认后选择日报。

### Frontmatter 必需字段

日报的 YAML frontmatter **必须包含以下所有字段**：

```yaml
---
title: {YYYY-MM-DD} 日报      # 标题
date: {YYYY-MM-DD}            # 日期
type: daily                    # 文档类型（固定值）
tags:                          # 标签列表
  - 日报
  - {项目标签}
week: {W}                      # ISO 周数（1-53）
total_hours: {N}               # 当日总工时
---
```

**注意**：`week` 字段使用 ISO 8601 周编号规则，3月27日属于第13周。

### 流程

1. **加载 obsidian-markdown skill**
2. **回顾对话**：提炼当天工作项
3. **项目标签策略（v1.1）**：
   - 先自动推断项目/模块标签
   - 仅在明显不确定时再向用户批量确认
   - 用户未回复时允许使用 `项目/未分类`，不阻断写入
4. **提炼工作项**：工作描述、问题与解决（如有）、工时
5. **推理反思与学习**：主动提炼经验
6. **检查已有日报并做合并去重**
7. **写入文件（UTF-8）**

## 周报生成

### 触发条件
用户明确要求周报，或在中触发确认后选择周报。

### Frontmatter 必需字段

周报的 YAML frontmatter **必须包含以下所有字段**：

```yaml
---
title: {YYYY} 第{W}周 周报     # 标题
type: weekly                   # 文档类型（固定值）
year: {YYYY}                   # 年份
week: {W}                      # ISO 周数（1-53）
date_range:                    # 周日期范围
  start: {YYYY-MM-DD}          # 周一日期
  end: {YYYY-MM-DD}            # 周五日期
tags:                          # 标签列表
  - 周报
  - {项目标签}
total_hours: {N}               # 本周总工时
---
```

### 流程

1. **加载 obsidian-markdown skill**
2. **确定周范围**：ISO 周（周一到周日）
3. **确定落盘目录**：周报放在“该周周四所在月份”目录
4. **读取日报**：遍历本周 7 天对应的日报文件
5. **缺失容错（v1.1）**：
   - 缺失某天日报时继续生成
   - 在周报增加 `> [!warning] 数据缺口` 列出缺失日期
6. **汇总提炼 + 合并去重**：按主题归并重复事项
7. **建立 wikilinks**
8. **写入文件（UTF-8）**

## 月报生成

### 触发条件
用户明确要求月报，或在中触发确认后选择月报。

### Frontmatter 必需字段

月报的 YAML frontmatter **必须包含以下所有字段**：

```yaml
---
title: {YYYY}年{MM}月 月报     # 标题
type: monthly                  # 文档类型（固定值）
year: {YYYY}                   # 年份
month: {MM}                    # 月份（01-12）
date_range:                    # 月日期范围
  start: {YYYY-MM-01}          # 月初日期
  end: {YYYY-MM-DD}            # 月末日期
tags:                          # 标签列表
  - 月报
  - {项目标签}
total_hours: {N}               # 本月总工时
---
```

### 流程

1. **加载 obsidian-markdown skill**
2. **确定月份范围**：当前月份或用户指定月份
3. **读取周报**：同月目录下全部周报
4. **缺失容错（v1.1）**：周报不全时继续生成，并在月报注明覆盖范围
5. **汇总提炼 + 合并去重**：按项目/模块归并
6. **建立 wikilinks**
7. **写入文件（UTF-8）**

## 年报生成

### 触发条件
用户明确要求年报，或在中触发确认后选择年报。

### Frontmatter 必需字段

年报的 YAML frontmatter **必须包含以下所有字段**：

```yaml
---
title: {YYYY}年 年度总结        # 标题
type: yearly                   # 文档类型（固定值）
year: {YYYY}                   # 年份
date_range:                    # 年日期范围（支持跨自然年）
  start: {YYYY-MM-DD}          # 起始日期
  end: {YYYY-MM-DD}            # 结束日期
tags:                          # 标签列表
  - 年报
  - {项目标签}
---
```

### 流程

1. **加载 obsidian-markdown skill**
2. **询问月份范围**：支持跨自然年
3. **读取月报**：按范围遍历
4. **必要时补充读取日报**
5. **按项目归类 + 合并去重**
6. **提取反思与成长**
7. **建立 wikilinks**
8. **写入文件（UTF-8）**

### 最小可交付（v1.1）
即使数据不足，也必须保证输出以下结构：
- 月报索引
- 工作总结（按项目至少 1 组）
- 问题与反思（可含 `%% TODO: 请补充 %%`）
- 下一年规划（至少 2 条草案，可含 TODO）

## 注意事项

- 生成文件前，务必先确保目录存在
- 追加内容时，先读取已有文件，再做结构化合并、去重、续号、统计更新
- 如果周报跨月，将周报放在该周周四所在月份目录（ISO 规则）
- 工时估算要合理：简单问答约 0.5h，复杂任务约 2-4h
- **信息安全**：不得写入代码片段、密钥、内部接口地址等敏感信息
- **Obsidian 兼容**：wikilinks 使用不含路径的文件名
- **Tags 规范**：tags 仅包含字母、数字、下划线、连字符、斜杠，且不能以数字开头
- **Obsidian 注释**：使用 `%% 注释 %%` 而非 HTML 注释
